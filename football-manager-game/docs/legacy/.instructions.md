### [Q5] AUSENCIA DE DETERMINISMO EN LA SIMULACIÓN
**Archivos**:
- `src/matchEngine.js`
- `src/gameEngine.js`

**Problema**:
Toda la simulación depende de `Math.random()`.
Guardar/cargar durante un partido produce resultados divergentes.

**Fix**:
- Implementar RNG seedable (Mulberry32 o equivalente)
- Guardar `seed` dentro de `liveMatch`
- Restaurar RNG al cargar save
- Reemplazar llamadas directas a `Math.random()`

**Implementación sugerida**:
```js
function mulberry32(a) {
  return function() {
    let t = a += 0x6D2B79F5;
    t = Math.imul(t ^ t >>> 15, t | 1);
    t ^= t + Math.imul(t ^ t >>> 7, t | 61);
    return ((t ^ t >>> 14) >>> 0) / 4294967296;
  };
}
```

### [Q6] MÚLTIPLES requestAnimationFrame ACTIVOS
**Archivo**: `src/matchVisualizer.js`

**Problema**:
Se detectaron múltiples llamadas a `requestAnimationFrame`.
Si `dispose()` no cancela el RAF previo, cada montaje agrega otro loop.

**Consecuencia**:
- FPS degradado
- CPU usage creciente
- Simulación acelerada accidentalmente
- Memory leak progresivo

**Fix**:
- Centralizar RAF loop
- Guardar ID:

```js
this._rafId = requestAnimationFrame(this._loop.bind(this));
```

- Cancelar en `dispose()`:

```js
if (this._rafId) {
  cancelAnimationFrame(this._rafId);
  this._rafId = null;
}
```

- Verificar que `create()/mount()` no creen múltiples loops simultáneos.
